# 咖啡厅在线点餐与消费积分系统

接口设计说明（V1.4）

> 本版本在 V1.3 基础上：
>
> - 补充了时间/金额字段格式与分页统一约定；
> - 扩展常用业务错误码说明，明确登录失败、库存不足、状态不允许、优惠券/积分不足等场景；
> - 补充退出登录、修改密码、订单详情、积分流水、优惠券等接口的示例结构；
> - 增加订单状态枚举、店员/管理员订单详情与仪表盘统计接口说明，使接口与页面设计更加一致；
> - 新增“会员价 + 购物车价格展示规则”说明，使商品列表、购物车、结算与订单明细在会员价计算上保持一致。

## 1 文档概述

1 文档概述

1.1 编写目的本接口设计说明文档用于规范“咖啡厅在线点餐与消费积分系统”的后端 REST API 接口定义，作为前后端协作、开发与测试的依据。本版本（V1.1）在 V1.0 的基础上，结合“朝暮”品牌设定，对积分（光阴值）、会员等级以及按时间段推荐商品等相关接口进行了扩展说明。

1.2 适用范围本接口文档适用于：

- 前端开发人员：基于本说明进行接口调用与页面功能实现；
- 后端开发人员：按照本说明实现各接口的业务逻辑；
- 测试人员：根据本说明编写接口测试用例与联调测试方案。

  1.3 接口风格与基础约定

- 接口风格：RESTful，使用 HTTP/HTTPS 协议。
- 传输格式：请求与响应均采用 JSON。
- 统一前缀：所有业务接口统一以 /api 作为前缀，例如 /api/auth/login。
- 认证方式：登录成功后，前端需在后续请求头中携带 Authorization: Bearer <token>。
- 时间格式：除特殊说明外，所有时间字段统一使用 `yyyy-MM-dd HH:mm:ss` 字符串表示，以服务器时区为准。
- 金额字段：所有金额字段（如 price、totalAmount、discountAmount、payableAmount、paidAmount 等）统一以“元”为单位，使用两位小数表示（例如 27.50）。

  1.4 统一响应格式除个别特殊情况外，所有接口统一采用如下响应结构：

{ "code": 0, "message": "success", "data": {...} }

- code：业务状态码，0 表示成功，非 0 表示失败。
- message：提示信息，成功时通常为 "success"，失败时为错误文案。
- data：具体业务数据对象或列表。

  1.5 统一错误码示例（可根据实际扩展）

- 0：成功
- 4001：参数校验错误
- 4010：未登录或登录已过期
- 4030：无访问权限
- 4040：资源不存在
- 4090：业务冲突（如库存不足、状态不允许操作）
- 5000：系统内部错误
- 常用业务错误码示例（可根据实际扩展）：
- 1001：用户名或密码错误。
- 2001：商品库存不足（如下单或调整库存时）。
- 2002：订单状态不允许当前操作（例如重复支付、重复取消、状态已变更）。
- 3001：优惠券不可用（已使用、已过期或不满足使用门槛）。
- 3002：积分不足，无法兑换优惠券。

  1.6 分页参数与返回结构约定

- 入参：如无特殊说明，分页查询统一使用 `page`（当前页码，从 1 开始）和 `size`（每页条数，默认 10，最大值可根据业务限制，例如 100）。
- 返回：分页接口的 data 字段统一为：

{ "total": 100, "page": 1, "size": 10, "list": [ /* 数据列表 */ ] }

- 说明：订单列表、积分流水列表、后台各类列表等分页接口均遵循该约定，除非在对应小节中另有说明。

## 2 认证与用户接口

2 认证与用户接口

2.1 用户注册接口

- 接口地址：/api/auth/register
- 请求方法：POST
- 权限要求：游客（未登录）可访问
- 功能说明：用户注册成为会员。

请求体示例：{ "username": "eudora", "password": "123456", "confirmPassword": "123456", "mobile": "13800000000" }

响应示例：{ "code": 0, "message": "注册成功", "data": { "userId": 1, "username": "eudora" } }

2.2 用户登录接口

- 接口地址：/api/auth/login
- 请求方法：POST
- 权限要求：游客可访问
- 功能说明：用户登录，返回访问令牌和用户基本信息。

请求体示例：{ "username": "eudora", "password": "123456" }

响应示例：{ "code": 0, "message": "登录成功", "data": { "token": "jwt-token-xxxx", "user": { "id": 1, "username": "eudora", "nickname": "Eudora", "role": "member", "level": 2, "levelName": "烈阳" } } }

说明：

- level 表示会员等级数值，如 1=晨曦,2=烈阳,3=晚霞,4=繁星；
- levelName 为后端根据 level 映射的展示名称，方便前端直接使用。
- username 字段可作为登录账号使用，前端界面可展示为“账号（可使用手机号）”，与注册时填写的账号字段保持一致。
- 登录失败时：当用户名不存在或密码错误，返回错误码 1001，并在 message 中给出友好提示。

  2.3 获取当前登录用户信息

- 接口地址：/api/user/me
- 请求方法：GET
- 权限要求：会员/店员/管理员需登录
- 功能说明：根据请求头中的 token 获取当前登录用户的信息。

响应示例：{ "code": 0, "message": "success", "data": { "id": 1, "username": "eudora", "nickname": "小爱喝拿铁", "mobile": "13800000000", "avatar": "https://example.com/avatar.png", "role": "member", "level": 2, "levelName": "烈阳" } }

2.4 用户退出登录（可选）

- 接口地址：/api/auth/logout
- 请求方法：POST
- 权限要求：会员/店员/管理员登录
- 功能说明：如需在服务端记录登出状态或实现 Token 黑名单，可实现该接口；若系统采用完全无状态 JWT，也可由前端删除本地 token，此接口可不实现，仅作为文档说明。

响应示例：{ "code": 0, "message": "退出成功", "data": null }

2.5 修改密码接口（可选）

- 接口地址：/api/user/change-password
- 请求方法：PUT
- 权限要求：会员登录
- 功能说明：用户在登录状态下修改密码，用于“个人中心-安全设置”页面。

请求体示例：{ "oldPassword": "123456", "newPassword": "abcdef", "confirmPassword": "abcdef" }

响应示例：{ "code": 0, "message": "修改成功", "data": null }

2.6 更新当前用户资料

- 接口地址：/api/user/me
- 请求方法：PUT
- 权限要求：会员登录
- 功能说明：修改当前登录会员的基本资料。

请求体示例：{ "nickname": "小爱喝拿铁", "mobile": "13800000001", "avatar": "https://example.com/avatar_new.png" }

响应示例：{ "code": 0, "message": "更新成功", "data": null }

6.1.1 会员等级价格计算规则（说明）

为体现“晨曦 / 烈阳 / 晚霞 / 繁星”四个会员等级的差异，系统对商品价格采用“原价 + 会员价基准值 + 等级梯度折扣”的统一规则：

- 若 memberPrice 为空或大于等于 price，则本次商品不启用会员折扣，四个等级统一使用 price 作为结算单价；以下梯度公式仅在 0 < memberPrice < price 时生效。
- 记：
  - P = 商品原价，对应字段 price；
  - Pm = 会员价基准值，对应字段 memberPrice，表示 Lv4 繁星会员在无其它优惠时可享受的价格；
  - L = 用户会员等级，取值 1~4（1=晨曦, 2=烈阳, 3=晚霞, 4=繁星）；
  - D = P - Pm，为原价与最高等级会员价之间的差额。
- 后端在创建订单时，根据当前登录用户的 level 计算本次结算价格：

  单品等级价公式：

  > priceLevel(L) = P - D * (L / 4)

- 说明：
  - 当 L = 1（晨曦），价格最接近原价，只享受少量折扣；
  - 当 L = 4（繁星），价格等于 memberPrice，即享受最大折扣；
  - 中间等级在原价与最高等级会员价之间按线性梯度平滑过渡；
  - 计算结果四舍五入保留两位小数。
- 订单生成时：
  - 计算后的单品等级价写入订单明细表的快照字段（如 product_price）；
  - subtotal_amount = priceLevel(L) × quantity；
  - 优惠券、积分抵扣等其它优惠在此基础上继续叠加计算。
- 安全性：
  - 前端仅做价格展示与预估，所有最终成交价以后端根据上述公式计算结果为准，防止前端被篡改导致价格异常。

## 10 会员价与购物车价格展示规则

10 会员价与购物车价格展示规则

本小节用于统一商品原价、会员价基准值、会员等级与购物车/结算/订单中的价格计算与展示规则，保证前后端在“会员价”相关的业务上行为一致。

10.1 字段与基础约定

- 商品价格字段：
  - `price`：商品原价，DECIMAL(10,2)，所有接口必填；
  - `memberPrice`：会员价基准值，DECIMAL(10,2)，可为 null。
- 会员价开启条件：
  - 当 `memberPrice` 为空（null）、小于等于 0 或大于等于 `price` 时，视为“未开启会员价”，所有会员等级统一按 `price` 计价；
  - 仅当 `0 < memberPrice < price` 时，才认为该商品开启会员价梯度规则。
- 用户会员等级：
  - `level`：整数 1~4；
  - 1=晨曦、2=烈阳、3=晚霞、4=繁星；
  - 当 level 超出范围（null、<1、>4）时，后端按 1 级（晨曦）处理。

10.2 统一价格计算公式（回顾）

当 `0 < memberPrice < price` 时，系统按以下公式计算不同会员等级下的单品结算价：

- 记：
  - `P` = `price`（商品原价）；
  - `Pm` = `memberPrice`（最高等级 Lv4 繁星在无其他优惠时可享受的单品价格基准）；
  - `L` = 用户会员等级（1~4）；
  - `D` = `P - Pm`（原价与最高等级会员价之间的差额）。

- 单品等级价公式：

  > priceLevel(L) = P - D * (L / 4)

- 结果处理：
  - 计算结果四舍五入保留两位小数；
  - 当 `memberPrice` 未开启（见 10.1）时，四个等级统一使用 `price` 作为结算单价，不套用上述公式。

10.3 各接口返回中的价格含义

为便于前端理解与使用，商品相关接口中与价格有关的字段含义统一约定如下：

1）前台商品接口（用户端）

- `/api/products`、`/api/products/{id}`、`/api/products/recommend`：
  - 均返回：
    - `price`：商品原价；
    - `memberPrice`：会员价基准（可选字段，缺失或为 null 表示未配置会员价）。
  - 前端可根据当前登录用户的 `level`，在 UI 上自行计算并展示“原价 + 当前等级价”（仅用于展示与预估），但**最终成交价以结算/订单接口中后端计算结果为准**。

2）管理后台商品接口

- `/api/admin/products`（列表）、`/api/admin/products/{id}`（详情）：
  - 同样返回 `price + 可选 memberPrice`；
  - 更新商品 `PUT /api/admin/products/{id}`：
    - 当前端关闭“会员价开关”时，会发送：`"memberPrice": null`，后端需将 `t_product.member_price` 更新为 `NULL`；
    - 当前端开启开关并填写数值时，发送具体的 `memberPrice` 数值，后端需在满足 `0 < memberPrice < price` 前提下保存；否则按业务可选择报错或视为未开启会员价。

10.4 购物车价格展示（/api/member/cart）

- 接口地址：`GET /api/member/cart`；
- 行为约定：
  - 后端会根据当前登录用户的 `level`，结合每个商品的 `price` 与 `memberPrice`，按 10.2 中的公式计算出本次购物车展示所用的单价；
  - 返回结构中的：
    - `unitPrice`：当前会员等级下的实际展示单价（已经应用会员价梯度折扣，如开启）；
    - `subtotal`：`unitPrice * quantity`，保留两位小数；
    - `summary.totalAmount`：为所有可用商品的 `subtotal` 之和。
- 等级变化：
  - 当用户会员等级发生变化时，再次请求 `/api/member/cart`，后端会基于最新等级重新计算 `unitPrice` 与 `subtotal`，前端无需自行缓存会员价。

10.5 结算与订单中的价格

1）结算预览 `/api/member/checkout/preview`

- 后端会再次根据当前用户等级 + 商品 `price/memberPrice` 重新计算所有行的单价与小计，不信任购物车中任何历史价格；
- 预览返回中的金额字段（itemsAmount、discountAmount、payAmount）均基于上述计算结果。

2）提交订单 `/api/member/checkout/submit`

- 提交订单时，后端会再次校验商品可用性与库存，并按同一会员价规则重新计算每一项的成交单价；
- 订单明细表快照字段：
  - `productPrice`：下单时该用户实际使用的单品单价（已包含会员价梯度折扣）；
  - `subtotalAmount`：`productPrice * quantity`，为行小计金额；
- 后续订单详情、支付结果、订单列表展示等接口，均应以订单明细中的快照价格为准。

10.6 前后端职责边界（给前端同学看的 TL;DR）

- 前端：
  - 可使用接口返回的 `price + memberPrice + level` 在商品列表、详情页和购物车中展示“原价 + 各等级价”，提升交互体验；
  - 但前端传给后端的价格字段仅用于展示或预估，**不参与最终成交价的可信计算**。
- 后端：
  - 是价格计算的单一可信来源，所有结算、优惠券判断、积分发放与订单明细快照中的价格，均由后端根据本节约定独立计算；
  - 不信任客户端传来的任何单价或小计；即使前端提交了价格信息，后端也会重新按规则计算并覆盖。
