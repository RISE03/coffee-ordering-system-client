# 咖啡厅在线点餐与消费积分系统
需求规格说明书（V1.1）
## 1 引言
1 引言

1.1 编写目的
本需求规格说明书用于描述“咖啡厅在线点餐与消费积分系统”的功能性需求和非功能性需求，为系统的设计、开发、测试和验收提供统一依据。本版本（V1.1）在 V1.0 的基础上，结合“朝暮”品牌设定，对积分（光阴值）、会员等级以及基于时间的动态主题与菜单推荐等特色需求进行了补充说明。

1.2 适用范围
本说明书适用于：
- 开发人员：根据本说明实现系统功能；
- 测试人员：根据本说明编写测试用例，执行功能测试与验收；
- 项目指导教师 / 评审专家：了解系统功能范围和特色设计；
- 运维人员：了解系统运行所需的基本业务规则。

1.3 术语与缩略语
- 会员：指使用系统进行在线点餐的普通用户。
- 店员：指在咖啡厅门店接单、制作饮品、处理取餐的操作人员。
- 管理员：指对商品、分类、订单、用户、积分规则等进行管理的后台用户。
- 光阴值：系统中对“积分”的品牌化称呼，用于体现用户在“朝暮”空间中累积的时间与陪伴。本需求中“积分”与“光阴值”在业务含义上等价。
- SRS：软件需求规格说明书（Software Requirements Specification）。

## 2 系统概述
2 系统概述

2.1 系统背景
随着移动互联网和外卖平台的发展，越来越多的咖啡厅希望通过线上渠道完成点餐、支付和会员运营。本系统以“朝暮（Dawn & Dusk）”为品牌核心，面向某咖啡厅场景，提供线上点餐、会员管理、积分（光阴值）和时间感知推荐等功能，帮助门店提高运营效率和用户体验。

2.2 系统目标
- 为会员提供便捷的在线点餐体验，包括商品浏览、购物车、下单、支付、订单查询与评价等。
- 为店员提供高效的订单处理工具，包括接单、制作、取餐核销等。
- 为管理员提供后台管理能力，包括商品、分类、用户、订单、积分规则和优惠券模板等。
- 通过积分（光阴值）、会员等级和优惠券等手段提升用户粘性与复购率。
- 通过动态主题（朝/暮）和时间段推荐菜单强化品牌氛围，提供兼具功能与情绪价值的使用体验。

2.3 用户角色与典型场景
（1）会员（Member）
- 在手机或 PC 浏览器中访问系统前台，查看“朝/暮”主题界面；
- 浏览商品、加入购物车、在线下单和支付；
- 查看订单状态、申请退款、完成评价；
- 查看光阴值（积分）和等级、兑换优惠券、使用优惠券。

（2）店员（Staff）
- 使用店员端界面登录系统；
- 查看待处理订单列表，执行接单、制作完成、取餐核销等操作；
- 在用户取餐时通过订单号或取餐码完成核销。

（3）管理员（Admin）
- 通过管理后台登录系统；
- 管理商品及分类、维护库存和上下架状态；
- 管理订单与退款审核；
- 配置积分规则与优惠券模板；
- 管理会员账号和店员账号；
- 可选：维护商品时间段配置，以支持按时间段推荐菜单。
## 3 功能性需求
3 功能性需求

3.1 角色与权限需求

3.1.1 角色类型
系统至少应支持以下三类角色：
- 会员（member）：可以访问前台点餐功能、个人中心、订单与积分等模块；
- 店员（staff）：可以访问店员端订单处理功能；
- 管理员（admin）：可以访问后台管理功能。

3.1.2 权限控制
- 未登录用户只能访问登录、注册页面以及部分商品浏览页面；
- 登录后根据角色显示不同的功能菜单和页面；
- 店员端与管理员端的接口应进行角色校验，防止越权访问。

3.2 会员端功能需求

3.2.1 用户注册与登录
- 系统应支持通过用户名 + 密码的方式注册成为会员。
- 注册时应进行基本校验（用户名唯一、密码长度等）。
- 支持会员通过用户名 + 密码进行登录。
- 登录成功后，系统应返回访问令牌和基本用户信息（包括等级信息）。

3.2.2 个人信息与账户管理
- 会员可在“个人中心”查看和修改昵称、头像、手机号等信息。
- 会员可查看自身角色和当前会员等级（晨曦/烈阳/晚霞/繁星）。

3.2.3 商品浏览与搜索
- 会员和未登录用户均可浏览商品列表。
- 支持按分类、关键字等条件搜索商品。
- 商品列表应展示商品名称、价格、缩略图和基础信息。
- 用户可查看商品详情，包括图片、详细介绍和价格。

3.2.4 购物车管理
- 已登录会员可将商品加入购物车。
- 会员可查看购物车列表，调整商品数量或移除商品。
- 会员可选择部分或全部购物车条目进行结算。

3.2.5 订单创建与支付
- 会员可基于购物车或立即购买创建订单。
- 下单时可选择取餐方式（自取 / 外卖），选择/填写取餐信息或收货地址。
- 系统根据商品价格、数量、优惠券等计算订单金额。
- 支持使用优惠券抵扣部分金额（如符合满减条件）。
- 本系统可采用“模拟支付”的方式，在系统内完成支付流程的状态变更。

3.2.6 订单查询与售后
- 会员可查看个人历史订单列表，支持按状态筛选（如待支付、制作中、已完成等）。
- 会员可查看订单详情，包括商品明细、金额、优惠、当前状态和状态变更记录。
- 对于未支付订单，会员可在允许时间内申请取消。
- 对于已支付但未制作或未取餐的订单，会员可在规则允许的情况下发起退款申请。

3.2.7 订单评价
- 会员可对已完成订单进行评价，包括评分（1~5 星）和文字评价。
- 每个订单原则上只允许评价一次。

3.2.8 积分（光阴值）与等级功能
- 每笔完成的订单可按照积分规则为会员累积一定数量的积分（光阴值）。
- 在前台展示时，积分统一命名为“光阴值”，例如“本次获得光阴值 +28”。
- 会员可在积分中心查看当前光阴值余额、累计获得和累计消费情况。
- 会员等级应根据累计消费金额或光阴值进行判定，至少包括：晨曦、烈阳、晚霞、繁星四个等级。
- 会员可在个人中心查看当前等级和距离下一等级的差距。

3.2.9 优惠券功能
- 会员可通过光阴值（积分）兑换优惠券模板生成的优惠券。
- 会员可查询自己持有的优惠券，查看有效期、使用条件和状态。
- 在创建订单时，会员可选择符合条件的优惠券进行抵扣。

3.3 店员端功能需求

3.3.1 店员登录
- 店员通过专用登录入口或通用登录入口登录系统，系统根据角色显示店员端页面。

3.3.2 订单处理
- 店员可查看待处理订单列表，支持按状态和时间过滤。
- 店员可对“已支付/待接单”订单执行接单操作，将订单状态更新为“制作中”。
- 制作完成后，店员可将订单状态更新为“待取餐”。
- 用户到店取餐时，店员可通过订单号或取餐码进行核销，将订单状态更新为“已完成”。

3.4 管理员端功能需求

3.4.1 管理员登录
- 管理员通过后台登录入口进入管理系统，需进行账号密码校验。

3.4.2 商品与分类管理
- 管理员可维护商品分类，包括新增、修改、删除、排序、启用/禁用。
- 管理员可维护商品，包括新增、修改、删除、上下架、库存调整等。

3.4.3 订单与退款管理
- 管理员可查询所有订单，支持按状态、用户、时间等条件筛选。
- 管理员可查看订单详情及状态变更记录。
- 对于用户发起的退款申请，管理员可审核并做出“同意退款”或“拒绝退款”的决策。

3.4.4 会员与店员管理
- 管理员可查看会员列表、查看会员详情、调整会员状态（启用/禁用）。
- 管理员可维护店员账号，包括新增店员、修改信息、重置密码、启用/禁用等。

3.4.5 积分规则与优惠券模板管理
- 管理员可配置或调整全局积分规则（如每消费 1 元获得多少光阴值）。
- 管理员可维护优惠券模板，包括满减门槛、减免金额、所需光阴值、有效期、启用/停用等。

3.4.6 系统配置与日志查询
- 管理员可维护系统配置（如系统公告、营业时间、主题相关说明等）。
- 管理员可按操作人、时间、类型查询操作日志，便于审计与问题追踪。

3.5 品牌与时间感相关特色需求（V1.1 新增）

3.5.1 动态主题（朝/暮模式）
- 系统应根据服务器时间或用户本地时间，实现“朝模式（浅色暖阳金）”与“暮模式（深色黛蓝）”的自动切换：
- 白天时间段自动应用 Dawn 主题；
- 夜间时间段自动应用 Dusk 主题。
- 前端界面右上角应提供显著的主题切换控件（如 ☀️ / 🌙 图标），用户可手动切换当前主题。
- 对于手动选择主题的用户，系统应记住其选择（例如存储于浏览器 localStorage），优先使用用户选择的主题。

3.5.2 时间段菜单推荐
- 系统应支持基于时间段的商品推荐，提供“朝·醒/午·憩/暮·微醺”等不同推荐区块：
- 06:00–11:00：优先推荐提神咖啡、早餐搭配等；
- 11:00–17:00：优先推荐手冲咖啡、拿铁、甜点等；
- 17:00–24:00：优先推荐气泡水、无酒精特调、晚安牛奶等。
- 后端应提供专门的推荐接口（如 /api/products/recommend），根据当前时间段返回对应区块标题（如“朝·醒推荐”）及推荐商品列表。
- 管理员端可选提供商品时间段配置管理功能，允许对某商品配置适用时间段及推荐优先级。

3.5.3 光阴值与等级展示
- 在积分中心和个人中心界面中，需使用“光阴值”作为对积分的前台展示名称。
- 前端需根据用户等级（level）展示相应名称及视觉元素，例如：
- 1：晨曦；2：烈阳；3：晚霞；4：繁星。
- 在订单完成页、积分变动提示等位置，可以通过文案（如“本次消费为您带来 +28 光阴值”）强化“时间与陪伴”的品牌概念。
## 4 非功能性需求
4 非功能性需求

4.1 性能需求
- 系统应支持至少单店日常运行场景下的并发访问，前台页面响应时间在正常网络环境下一般不超过 3 秒。
- 商品列表、订单列表等分页接口应支持分页查询，并避免一次返回过多数据造成加载延迟。
- 后续如需扩展多门店或更大并发量，可以通过增加缓存（如 Redis）、读写分离等方式提升性能。

4.2 安全性需求
- 用户密码在数据库中必须采用安全的哈希算法（如 BCrypt）进行加密存储。
- 接口访问需基于 JWT 等机制进行身份认证，所有需要登录的接口必须校验访问令牌。
- 管理员、店员等后台角色的高危操作（如删除商品、退款处理）应记录操作日志。
- 对关键输入进行服务端校验，防止 SQL 注入和常见 Web 安全问题。

4.3 可用性与易用性
- 前台界面应简洁、易懂，重要操作（如下单、支付、取餐）流程清晰。
- 动态主题切换应平滑、不过分影响阅读，夜间模式应保证足够的对比度与可读性。
- 对重要操作提供适当的确认提示和错误提示信息。

4.4 可维护性与可扩展性
- 系统采用前后端分离架构，前端 Vue、后端 Spring Boot，可分别独立升级与部署。
- 数据库设计遵循规范的命名与字段定义，支持通过 Flyway 等工具进行版本迁移和回滚。
- 核心业务如点餐、积分、优惠券等模块相互独立，方便后续按模块扩展或重构。
- 时间段推荐、动态主题等特色功能以配置和前端样式为主，不影响核心业务流程，可根据实际运营情况调整或关闭。
## 5 其他需求
5 其他需求

5.1 日志与监控
- 系统应记录关键业务事件日志，如用户登录、订单状态变更、库存调整、积分变更等。
- 日志内容需包含操作人、时间、操作类型、目标对象等信息，便于后期追踪。

5.2 部署与运行环境（建议）
- 后端运行环境：JDK 17 或 21，Spring Boot 3.x，MySQL 8.x，Redis（可选）。
- 前端运行环境：Node.js、Vue 3 + Vite、Element Plus 等。
- 具体部署方式可根据学校实验环境或个人服务器环境灵活选择。

5.3 版本控制与演进
- 本文档为 V1.1 版本，主要是在 V1.0 基础上补充了品牌与时间感相关的特色需求。
- 后续如需增加多门店支持、第三方支付（如支付宝、微信支付）对接等功能，可在 V2.x 版本中扩展说明。
